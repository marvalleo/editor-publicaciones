## 1.6 — Panel de capas (izquierda)

**Objetivo:** lista de capas con visible/bloqueo, subir/bajar (orden z), duplicar,
eliminar, renombrar — según `CLAUDE.md` §7, Tarea 1.6.

**Ubicación:** nueva sección "CAPAS" en el panel izquierdo (`_build_left`), entre
el selector de formato y los campos de texto. Se mantiene todo en `app.py` (no se
extrae `left_panel.py` todavía, mismo criterio usado en 1.1–1.5 de mantener un solo
archivo hasta que el plan lo pida explícitamente).

**Orden de la lista:** de arriba hacia abajo, de mayor a menor `z` (la capa que se
dibuja al frente aparece primero en la lista — convención estándar de editores de
capas).

**Cada fila muestra:**
- Ícono/botón ojo (👁/🙈) → toggle `visible`.
- Ícono/botón candado (🔒/🔓) → toggle `locked`.
- Nombre de la capa (`layer.name`). Click selecciona la capa (mismo `_set_selected`
  que ya usa el click en el canvas). Doble-click activa edición inline (un
  `tk.Entry` reemplaza el label, `Enter`/`FocusOut` confirma, `Escape` cancela).
- Fila resaltada (fondo `ACCENT`-ish o borde) si `layer is self._selected`.
- Botones ▲/▼ para subir/bajar en el orden z (intercambia `z` con el vecino
  correspondiente).
- Botón "duplicar" (⧉): clona la capa completa (misma clase, nuevo `id` vía
  `_short_id()`, `z` = `max(z existentes) + 1`, nombre con sufijo " (copia)"),
  la agrega a `self.slide.layers` y la selecciona.
- Botón "eliminar" (🗑): quita la capa de `self.slide.layers`. Si era la
  seleccionada, deselecciona. Sin capas mínimas obligatorias — se puede borrar
  cualquier capa, incluida la foto (el render ya soporta lienzo sin foto, capa
  transparente).

**Refresco:** un método `_refresh_layers_list()` reconstruye las filas (mismo
patrón que `_build_property_panel`: destruye hijos del contenedor y los vuelve a
crear). Se llama después de: agregar/duplicar/eliminar/reordenar/renombrar capa,
cambiar `visible`/`locked` desde el panel de capas, y cada vez que cambia la
selección (`_set_selected`) para resaltar la fila activa. También sincroniza el
panel de propiedades (`_build_property_panel`) cuando la selección cambia desde
la lista.

**Nota para 1.7:** cada una de estas acciones (mover/eliminar/duplicar/reordenar/
renombrar/toggle) es un punto de integración con el `CommandStack` de la Tarea 1.7
— se implementan ya pensando en pasar por `self.commands.push(...)` en vez de
mutar directamente, para no tener que reescribirlas.

## 1.7 — Undo/Redo

**Objetivo:** `CommandStack` para mover, redimensionar, cambios de propiedad,
agregar/eliminar/duplicar/reordenar capa. Atajos `Ctrl+Z`/`Ctrl+Y`.

**Módulo nuevo:** `dcpub/commands.py`.

**Patrón:** Command clásico con dos tipos de comando:

1. `PropertyChangeCommand(target, attr, old_value, new_value)` — cubre **todo**
   cambio de un único atributo simple de una capa: mover (x/y), redimensionar
   (size/w/h), opacidad, zoom/offset de foto, toggle visible/locked, centrar
   rápido. `undo()` hace `setattr(target, attr, old_value)`, `redo()`/`execute()`
   hace `setattr(target, attr, new_value)`. Una sola clase cubre todos estos casos
   porque todos son "restaurar un valor anterior a un atributo".
2. Comandos estructurales, uno por operación, cada uno opera sobre
   `self.slide.layers` (lista) directamente:
   - `AddLayerCommand` (usado por duplicar): `undo()` remueve la capa,
     `execute()` la vuelve a insertar en la misma posición.
   - `DeleteLayerCommand`: guarda `(layer, index)` al crear; `undo()` reinserta en
     `index`, `execute()` remueve.
   - `ReorderLayerCommand`: guarda los `z` antes/después del swap; `undo()`/
     `execute()` restauran cada lado.
   - `RenameLayerCommand`: caso particular de `PropertyChangeCommand` sobre
     `attr="name"` — no necesita clase propia, se reusa `PropertyChangeCommand`.

**`CommandStack`:**
```python
class CommandStack:
    def __init__(self): self._undo = []; self._redo = []
    def push(self, command):
        command.execute()
        self._undo.append(command)
        self._redo.clear()
    def undo(self):
        if not self._undo: return False
        cmd = self._undo.pop(); cmd.undo(); self._redo.append(cmd); return True
    def redo(self):
        if not self._redo: return False
        cmd = self._redo.pop(); cmd.execute(); self._undo.append(cmd); return True
    def clear(self): self._undo.clear(); self._redo.clear()
```
`push()` ya ejecuta el comando — quien llama construye el comando con los valores
viejo/nuevo y lo empuja, no lo ejecuta a mano antes.

**Granularidad de "un gesto = un comando" (evita spamear el stack):**
- **Drag en canvas** (`_on_drag`/`_on_release`): captura el valor `(x,y)` inicial
  en `_on_press` (antes de empezar a arrastrar), aplica los cambios en vivo
  durante `_on_drag` **sin pasar por el stack** (igual que ahora), y en
  `_on_release` empuja **un solo** `PropertyChangeCommand` comparando el valor
  inicial contra el final (si cambió; si el usuario soltó en el mismo lugar, no
  se empuja nada).
- **Resize con handles**: mismo criterio — valor inicial capturado en
  `_start_resize`, un solo comando en `_on_release`.
- **Sliders** (`_slider`/`_on_slider`): el valor inicial se captura en
  `<ButtonPress-1>` sobre el `ttk.Scale` (nuevo binding), y se empuja un comando
  en `<ButtonRelease-1>` comparando contra el valor final. Los `_on_slider`
  intermedios (mientras se arrastra el thumb) siguen aplicando el cambio en vivo
  sin pasar por el stack, igual que el drag del canvas.
- **Entries numéricos** (`_on_entry_commit`): cada `Enter`/`FocusOut` ya es un
  evento discreto — un comando por commit.
- **Nudge por teclado**: cada pulsación de flecha ya es discreta — un comando por
  pulsación (no se agrupan pulsaciones consecutivas; es el mismo criterio que
  "cada acción de edición" del criterio de aceptación de Fase 1).
- **Toggle visible/locked, centrar rápido**: un comando por click de botón.
- **Agregar/duplicar/eliminar/reordenar/renombrar capa** (panel de capas, 1.6):
  un comando estructural por acción.

**Atajos:** `Ctrl+Z` → `self.commands.undo()`; `Ctrl+Y` **y** `Ctrl+Shift+Z` →
`self.commands.redo()` (ambos atajos son convención común, se agregan los dos).
Después de undo/redo: refrescar `_sync_sliders`, `_build_property_panel`,
`_refresh_layers_list`, `_render_now` — el estado del modelo cambió por fuera del
flujo normal de UI.

**Qué NO entra al stack:** cambios de formato (`_on_format_change`) y "Restablecer
posiciones" (`_reset`) quedan **fuera** del undo/redo en esta tarea — son
operaciones que reemplazan el proyecto/slide entero, no atributos de una capa
puntual, y el plan no las menciona explícitamente en el criterio de aceptación
("deshacer y rehacer cualquier **acción de edición**"). Se documenta como
limitación conocida, no como bug.

## 1.8 — Guardar / abrir proyecto

**Objetivo:** `ProjectManager` guarda/carga `.json` con rutas relativas, aviso de
cambios sin guardar.

**Módulo nuevo:** `dcpub/project_io.py`, clase `ProjectManager` con dos funciones
puras (sin tocar Tkinter, testeables directamente):

```python
def save(project: Project, path: Path) -> None:
    """Serializa el proyecto a JSON en `path`, con las rutas de imagen (photo/logo)
    reescritas a relativas respecto a la carpeta de `path` cuando sea posible."""

def load(path: Path) -> Project:
    """Carga un Project desde JSON en `path`, resolviendo las rutas relativas de
    imagen contra la carpeta de `path` a rutas absolutas utilizables."""
```

**Rutas relativas:** al guardar, para cada `PhotoLayer.src`/`LogoLayer.src` no
vacío, se intenta `Path(src).resolve().relative_to(path.parent.resolve())`; si
falla (`ValueError`, por estar en otra unidad/fuera del árbol), se guarda la ruta
absoluta tal cual. Al cargar, cualquier `src` que **no** sea absoluto
(`Path(src).is_absolute()` es `False`) se resuelve como
`(path.parent / src).resolve()`.

**Integración en `app.py`:**
- `self._project_path: Path | None = None` (ruta del `.json` actual, `None` si es
  un proyecto nuevo sin guardar todavía).
- `self._dirty: bool = False` — se pone en `True` en cada `self.commands.push(...)`
  exitoso (hook simple: `CommandStack` acepta un callback `on_change` opcional
  invocado en `push`/`undo`/`redo`; `App` le pasa `lambda: setattr(self, "_dirty", True)`).
  Se resetea a `False` justo después de guardar o abrir.
- Botones nuevos en el panel izquierdo: "💾 Guardar proyecto" (si `_project_path`
  es `None`, se comporta como "Guardar como…" pidiendo ubicación con
  `filedialog.asksaveasfilename(defaultextension=".json", filetypes=[("Proyecto", "*.json")])`;
  si ya hay `_project_path`, sobreescribe directo) y "📂 Abrir proyecto"
  (`filedialog.askopenfilename`, mismo filtro).
- Al abrir: si `self._dirty`, preguntar primero con
  `messagebox.askyesnocancel("Cambios sin guardar", "¿Guardar los cambios antes de abrir otro proyecto?")`
  (Sí = guardar y continuar, No = descartar y continuar, Cancelar = abortar la
  apertura). Mismo diálogo al cerrar la ventana
  (`self.protocol("WM_DELETE_WINDOW", self._on_close)`).
- Tras abrir: `self.project`/`self.slide` se reemplazan, `self.commands.clear()`
  (el historial de undo no cruza proyectos), `self._selected = None`,
  `self.v_photo`/`self.v_title`/`self.txt_title`/`self.v_sub`/`self.txt_desc`/
  `self.v_icon` se repueblan desde las capas cargadas (hoy el contenido de texto
  vive en variables de Tkinter, no en el modelo — ver nota de alcance de la Tarea
  1.5; esta tarea no cambia esa arquitectura, solo sincroniza esas variables al
  cargar/guardar).
- Título de la ventana: refleja el nombre de archivo actual y un `*` si hay
  cambios sin guardar (`self.title(f"... — {nombre}{'*' if self._dirty else ''}")`),
  actualizado en cada cambio de `_dirty`.

**Nota de alcance:** como el contenido de texto (título/subtítulo/descripción/
ícono/ruta de foto) vive en variables Tkinter y no en los campos `text`/`icon`/
`src` del modelo (decisión ya tomada en 1.1–1.5), `ProjectManager.save()` necesita
que quien la llama sincronice esos valores a las capas **antes** de serializar
(mismo patrón que ya usa `_build_layers()` para armar el dict de render). Se
agrega un método `_sync_text_to_layers()` en `app.py` que hace esa copia
puntualmente antes de guardar, sin tocar la arquitectura existente.

## 1.9 — Exportar

**Objetivo:** exportar a PNG (y JPG opcional) en resolución real, elegir carpeta
destino, crear `publicaciones/` si no existe, nombrado por proyecto + fecha/hora.

**Módulo nuevo:** `dcpub/exporter.py`, clase `Exporter` con una función pura:

```python
def export(project: Project, layers: list[dict], font_manager: FontManager,
           dest_dir: Path, fmt: str = "png", max_side: int = 2400) -> Path:
    """Renderiza `layers` a resolución real (lado mayor = max_side, ≥ formato
    nativo) vía compose(), guarda en `dest_dir` con nombre
    "<project.name-slugificado>_<YYYYmmdd_HHMMSS>.<png|jpg>", crea `dest_dir` si
    no existe, y devuelve la ruta final."""
```

`fmt` acepta `"png"` (con transparencia si corresponde) o `"jpg"` (se aplana
sobre fondo blanco antes de guardar, igual que hace hoy `_generate` con
`.convert("RGB")`). El nombre de proyecto se slugifica (espacios → `_`,
caracteres no alfanuméricos removidos) para que sea un nombre de archivo válido
en Windows.

**Integración en `app.py`:** se reemplaza el botón único "💾 Generar y guardar"
por un flujo con opciones mínimas:
- Combobox de formato de salida: PNG (default) / JPG.
- Botón "📁 Elegir carpeta…" (opcional; si no se elige, usa `OUTPUT_DIR` =
  `publicaciones/` junto al script, como hoy).
- Botón "📤 Exportar" que llama a `Exporter.export(...)` con
  `self._sync_text_to_layers()` + `self._build_layers()` ya aplicado, muestra
  éxito/error igual que la implementación actual de `_generate` (mismo patrón de
  `messagebox`/`v_status`).

**Compatibilidad:** la función vieja `_generate()` se reemplaza por esta nueva
ruta; no queda código duplicado. El criterio de aceptación de Fase 1 ("exportar
PNG idéntico a la vista previa") sigue cumplido porque `Exporter.export` usa el
mismo `compose()` que ya se usa para preview, solo a resolución mayor — sin
cambios al motor de render.
